# ShulSearch v1 — Technical Verification Findings

Consolidated review of four research tracks against the ShulSearch v1 design (Next.js App
Router + shadcn on Firebase App Hosting; Firestore shul DB; Firebase Auth Google-gated admins;
anonymous searchers; Zillow-via-RapidAPI listings behind a swappable `ListingsProvider`;
haversine radius filter + Google walking-time on shown results; OSM+Places-seeded shul DB
across ~11 US metros; denomination as a first-class filter). Date of record: 2026-07-03.
Generated by the `shulsearch-tech-verify` workflow (5 agents, ~184k tokens).

**Headline:** Three of the four subsystems (listings pipeline, App Hosting, Google Maps cost
pattern) are **validated as designed** with concrete refinements. The **shul-data-sourcing
subsystem is the one that must change** — Google Places cannot legally be the durable store,
and the stated "Places + OSM hybrid" under-covers exactly the Haredi metros ShulSearch targets
while leaving denomination (a first-class filter) largely unfilled.

---

## 1. Zillow Listings via RapidAPI (bounding-box pipeline)

**Bottom line:** The bounding-box → haversine → walking-time pipeline is **validated and can
stay as designed**. A native lat/lng map-bounds search endpoint exists, so the primary path
does **not** need to degrade to city-string search. There is **no official Zillow listings API
in 2026** — every "Zillow API" on RapidAPI is an unofficial scraper — so the swappable
`ListingsProvider` + caching + failover is load-bearing, not optional.

**Concrete facts**
- No official option. Zillow retired its public Web Services API in Sept 2021; the only official
  route is Bridge Interactive (~$500+/mo, weeks-long MLS-gated approval).
- **PRIMARY — OpenWeb Ninja "Real-Time Zillow Data"** (RapidAPI vendor
  `letscrape-6bRBa3QguO5/real-time-zillow-data`). Endpoints: `/search`,
  **`/search/coordinates` (explicit map bounds N/S/E/W lat/lng)**, `/search/polygon`,
  `/search/url`, `/property`. Filters: `status` (ForSale/ForRent/Sold), price range, beds/baths
  min, `home_type`. Each listing returns `zpid`, address, **lat/lng**, price, beds, baths, sqft,
  homeType, status, Zestimate + rent Zestimate, days on Zillow, broker, photos, Zillow link.
- **OpenWeb Ninja pricing:** Free = 100 req/mo; Pro = **$25/mo for 10,000 req** (10 req/s);
  Ultra = $75/mo/50k; Mega = $150/mo/200k; PAYG = **$0.005/request**.
- **FALLBACK — apimaker "Zillow.com"** (`zillow-com1.p.rapidapi.com`): most-subscribed. Core
  `/propertyExtendedSearch` is location-string; bounds only via `/searchByUrl` (mapBounds URL) —
  more fragile.
- ~500-result Zillow map cap is **irrelevant for a walkable radius** (usually 1 page). API
  returns a **rectangle**, so the post-query **haversine circle filter is required**.

**What to do:** first `ListingsProvider` = OpenWeb Ninja `/search/coordinates`; apimaker as the
second provider behind the same interface. Pipeline: bounds query → haversine circle filter →
walking-time on survivors only. Server-side cache (rounded bounds + filters, ~15–60 min TTL),
retry-with-backoff, provider health check/failover, nightly canary.

Sources: openwebninja.com/api/real-time-zillow-data · rapidapi.com/letscrape-6bRBa3QguO5/api/real-time-zillow-data · rapidapi.com/apimaker/api/zillow-com1

## 2. Firebase App Hosting (Next.js App Router)

**Bottom line:** App Hosting (GA since April 2025) is the right target and **fully compatible**.
Full SSR on managed Cloud Run + CDN, GitHub push-to-deploy, `apphosting.yaml`, native Secret
Manager. Admin SDK auto-inits via Application Default Credentials (no service-account JSON);
client SDK auto-configures from injected `FIREBASE_WEBAPP_CONFIG`. **One hard constraint: the
Admin SDK cannot run in Next.js Edge middleware** — session verification must live in Node.

**Concrete facts**
- Requires **Blaze** plan. GitHub is the **only** supported git provider. Compute = managed
  Cloud Run + CDN.
- **`apphosting.yaml`** `runConfig` defaults: `minInstances` 0, `maxInstances` 100,
  `concurrency` 80, `cpu` 1, `memoryMiB` 512.
- Env vars: `{variable, value, availability:[BUILD,RUNTIME]}`. **`NEXT_PUBLIC_*` are inlined at
  build time → MUST include `BUILD` availability** or they never reach the browser bundle.
- **Secrets:** Cloud Secret Manager, referenced as `{variable, secret}`. Create/grant with
  `firebase apphosting:secrets:set` / `:grantaccess`.
- **Admin SDK:** zero-arg `initializeApp()` via ADC; wrap init in `import 'server-only'`
  singleton. **Client SDK:** `FIREBASE_WEBAPP_CONFIG` auto-injected (public, safe to expose).
- **Session-cookie auth:** client Google sign-in → ID token → route handler →
  `createSessionCookie(idToken, {expiresIn})` → httpOnly+secure+sameSite cookie; guard with
  `verifySessionCookie(cookie, true)`. **Admin SDK does NOT run in Edge middleware** — verify in
  a server component / route handler / server action. Gate authz with custom claim `admin:true`.
- Pin Next.js to **15.x** ("active"); Node **20+** matching `package.json` engines.

**Refinements:** (1) `/admin` verification in Node runtime, not middleware; (2) split Maps into
two keys (server secret + optional browser key); (3) every `NEXT_PUBLIC_*` gets `BUILD`
availability; (4) grant compute service account Firestore + Secret Manager Secret Accessor.

Sources: firebase.google.com/docs/app-hosting/configure · /frameworks-tooling · /docs/auth/admin/manage-cookies

## 3. Google Maps Platform (Places New, Routes walking, Geocoding, pricing)

**Bottom line:** The **"walking time on shown results only" plan is exactly right — keep it.**
The universal $200/mo credit was retired March 1, 2025 → **per-SKU free caps that do NOT pool.**
Seeding and maps are effectively free at v1 volume; the **only line item that grows with
traffic is the walking-distance matrix**, hitting a soft cliff around **~500 searches/month.**
Google exposes **no denomination data** (only a `synagogue` place type).

**Concrete facts**
- Per-SKU free caps: Essentials 10,000 / Pro 5,000 / Enterprise 1,000 events/mo, reset the 1st
  at 00:00 PT, **not pooled**.
- **Places (New)** supports place type **`synagogue`**; **NO denomination field**. Text Search
  Pro $32/1000 (5,000 free) basic fields; **IDs-only = free**. Place Details Essentials $5/1000.
  Billed at the **highest SKU any requested field maps to**.
- **Walking distance:** Distance Matrix is **Legacy**; use **Routes API `computeRouteMatrix`**.
  Bills **per element (origins × destinations)**, Essentials **$5/1000 (10,000 free)**. 1 shul ×
  20 listings = 20 elements = one call.
- **Autocomplete** with session token = free keystrokes. **Dynamic Maps** $7/1000 (10k free).
- **`@vis.gl/react-google-maps`** v1.0 (OpenJS, Google-seeded) — de-facto recommended wrapper.
- **Rough v1 cost:** ~500 searches/mo ≈ **$0**; ~1,000 ≈ **$50**; ~2,000 ≈ **$150**. Seeding ≈ $0.

**Refinements:** (1) the walking matrix is the only Google cost that scales — plan for the
~500-free-search/mo cliff; (2) add a walking-time cache (shulId + rounded listing coords) + hard
15–20 result cap; (3) use Routes `computeRouteMatrix` with a `duration`+`distanceMeters` field
mask; (4) denomination is NOT an API field; (5) shul picker = Firestore search, not Autocomplete.

Sources: developers.google.com/maps/billing-and-pricing/pricing · /documentation/places/web-service/place-types · /documentation/routes/compute_route_matrix

## 4. Shul (Synagogue) Data Sourcing across ~11 US metros

**Bottom line — this is the subsystem that must change.** The stated "Google Places + OSM,
denomination first-class" is insufficient on two fronts. **Legal:** Google's Places ToS forbids
persisting any Places content except `place_id`, so Google cannot be the durable Firestore store
— only a runtime lookup keyed by a stored `place_id`. **Coverage:** the only freely-storable
source (OSM, ODbL) materially under-covers exactly the dense Orthodox/Haredi metros ShulSearch
centers on, and OSM `denomination` is filled on only ~13% of Jewish objects.

**Concrete facts**
- **OSM query (ODbL, storable w/ attribution):**
  `nwr["amenity"="place_of_worship"]["religion"="jewish"]({{bbox}}); out center tags;` (widen
  with `building=synagogue`).
- **OSM global:** 15,037 objects `religion=jewish`, only 49% places of worship, 36% cemeteries;
  `denomination` on only **13.3%**.
- **Live per-metro OSM Jewish places of worship (total / w/ denomination):** Brooklyn 185/51,
  Queens 148/33, Five Towns 26/3, Teaneck 11/2, **Lakewood 9/6**, Monsey 45/21, Baltimore 22/3,
  **LA core 23/1 (4%)**, Chicago W. Rogers Park 36/34 (94%), Miami Beach 14/3.
- **Under-count reality:** GoDaven lists **6,300+ Orthodox minyanim**; Lakewood alone has 100+;
  Boro Park/Flatbush have hundreds of shtieblach. OSM misses the **basement/storefront shtiebel
  layer** that dominates Haredi metros.
- **Google Places:** better named-congregation coverage, `synagogue` type, but **no
  denomination** and **ToS forbids persisting content except `place_id`** (refetch live;
  "Google Maps" attribution required off-map).
- **Restricted human-reference sources (do NOT bulk-scrape):** GoDaven (6,300+ minyanim; ToS
  prohibits copying/redistribution/commercial use; filters by nusach Ashkenaz/Sefard),
  Chabad.org locator (~3,500+ US centers; name token "Chabad" makes denomination trivial).
- **Authoritative denomination by affiliation:** URJ ~819 Reform, USCJ ~600 Conservative, OU +
  GoDaven for Orthodox. Name+geo cross-reference = highest-confidence signal.
- **Storable/open:** OSM = ODbL 1.0 (attribution + share-alike on derived DB). Wikidata = CC0
  (fully storable). ARDA RCMS 2020 = county-level counts (sanity check only).
- **Dedup recipe:** normalize names (strip diacritics/punctuation, expand abbreviations, unify
  Beth/Beis/Bais/Beit, drop generic stopwords) → geoblock (~75–120 m dense, ~150 m else) → match
  if fuzzy ≥~0.85 OR identical housenumber+street OR same website/phone → merge keeping OSM
  canonical + attach Google `place_id` → flag 0.70–0.85 for admin review.

**Required changes:** (1) durable records from **OSM (ODbL) + Wikidata (CC0) + first-party
curation**, Google reduced to stored `place_id` + live lookup; (2) **denomination pipeline** =
movement-directory cross-reference + name heuristics + admin curation, normalized to a fixed
enum with provenance/confidence; (3) **admin curation is a first-class part of v1 seeding**, not
an afterthought — to close the Haredi-metro gaps.

Sources: wiki.openstreetmap.org/wiki/Tag:religion=jewish · taginfo.openstreetmap.org · developers.google.com/maps/documentation/places/web-service/policies · godaven.com/terms-of-service · reformjudaism.org/find-a-congregation

## Blocking issues (break the design as written)

1. **Google Places cannot be the durable stored shul DB.** ToS forbids persisting Places content
   except `place_id`. Restructure so durable records come from OSM + Wikidata + first-party
   curation, Google reduced to a stored `place_id` + live lookup.
2. **Denomination (a first-class filter) is not supplied by the stated sources.** Google has no
   denomination field; OSM `denomination` is only ~13% filled. Needs a dedicated denomination
   pipeline (movement directories + name heuristics + admin curation, normalized enum + provenance).
3. **OSM materially under-covers the exact target metros** (Lakewood 9, LA 23, Teaneck 11) and
   misses the shtiebel layer. A first-class admin-curation layer must be part of v1 seeding, or
   the radius search is unusable in the dense Orthodox metros.

## Recommended design changes (non-blocking)

1. OpenWeb Ninja `/search/coordinates` = first `ListingsProvider`; apimaker `zillow-com1` = fallback.
2. Keep the haversine circle filter; ~500-cap box-tiling only as a guarded edge case.
3. Server-side listings cache + retry/backoff + provider health check/failover + nightly canary.
4. Routes API `computeRouteMatrix` (WALKING, `duration`+`distanceMeters` mask) instead of Legacy
   Distance Matrix.
5. Walking-time cache (shulId + rounded listing coords) + hard 15–20 result cap.
6. `/admin` session verification in Node runtime (not Edge middleware); custom claim `admin:true`.
7. Two Google Maps keys: server-only secret (Routes) + optional referrer-restricted browser key.
8. Secrets in Cloud Secret Manager (RUNTIME); every `NEXT_PUBLIC_*` gets `BUILD` availability.
9. Admin SDK zero-arg ADC init in `server-only` singleton; grant compute SA Firestore + Secret
   Accessor; pin Next.js 15.x, Node 20+.
10. Shul picker = Firestore search, not Google Autocomplete.
11. Seed with a minimal Places field mask; store `placeId` + lat/lng at seed time; adopt
    `@vis.gl/react-google-maps` v1.
12. Dedup + attribution pipeline; keep "© OpenStreetMap contributors" on stored data; decide if
    ODbL share-alike on the derived DB is acceptable or lean more on CC0 Wikidata + curation.
13. Cloud Billing budgets/alerts; restrict Maps key by referrer/API; plan the walking-matrix cost
    curve (~500/mo ≈ $0, ~1,000 ≈ $50, ~2,000 ≈ $150).
